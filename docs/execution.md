# Execution State

Every task execution is captured in a structured log — from the task itself down to individual LLM requests and tool calls. The `TaskStateStore` is the single source of truth for execution state.

## Execution Hierarchy

```
TaskLog                          ← one per task execution
  ├── id, name, plan, params
  ├── status, createdAt, completedAt
  ├── tokenUsage (aggregate)
  └── steps: Map<String, StepLog>
        └── StepLog              ← one per plan step
              ├── id, stepName
              ├── status, output, createdAt, completedAt
              ├── checkpoint (if HITL)
              ├── tokenUsage (aggregate)
              └── runs: List<RunLog>
                    └── RunLog   ← one per agent execution attempt
                          ├── id, index, agentName
                          ├── tokenUsage (aggregate)
                          └── turns: List<TurnLog>
                                └── TurnLog  ← one LLM round-trip
                                      ├── id, index
                                      ├── startedAt, completedAt
                                      ├── messageId, request (LlmRequest)
                                      ├── responseId, response (LlmResponse)
                                      └── toolResults: List<ToolResult>
```

A **step** can have multiple **runs** (retries after HITL rejection). A **run** has multiple **turns** (LLM call → tool calls → next LLM call). A **turn** captures one request-response cycle plus any tool results.

## IDs

Every object in the hierarchy has an 8-character hex ID generated by `IdGenerator.newId()`. IDs are created by the caller (TaskRunner, SmacAgentRunner) and passed into the store — the store does not generate IDs.

```
TaskLog  "a3cc1bf2"
  └── StepLog  "7e4f0a21"
        └── RunLog  "b8d31c09"
              └── TurnLog  "f1a20e5b"
```

## TaskStateStore

The store interface uses granular mutation methods mirroring the execution hierarchy:

```java
public interface TaskStateStore {

    // Task lifecycle
    void taskStarted(String taskId, String taskName, Plan plan, Map<String, String> params);
    void taskStarted(String taskId, String taskName, Plan plan, Map<String, String> params,
                     String parentTaskId, String parentStepId, int iterationIndex);
    void taskCompleted(String taskId, TaskStatus status);

    // Step lifecycle
    void stepStarted(String taskId, String stepId, String stepName);
    void stepCompleted(String taskId, String stepId, TaskStatus status, String output);

    // Run lifecycle
    void runStarted(String taskId, String stepId, String runId, String agentName);
    void runCompleted(String taskId, String runId);

    // Turn lifecycle
    void turnStarted(String taskId, String runId, String turnId);
    void turnCompleted(String taskId, String turnId);

    // Message level (within a turn)
    void messageSent(String taskId, String turnId, LlmRequest request);
    void responseReceived(String taskId, String turnId, LlmResponse response);

    // Tool call level (within a turn)
    void toolCallStarted(String taskId, String turnId, ToolCall toolCall);
    void toolCallCompleted(String taskId, String turnId, ToolResult toolResult);

    // HITL level (within a step)
    void hitlNotified(String taskId, String stepId, HitlCheckpoint checkpoint);
    void hitlResponded(String taskId, String stepId, HitlResponse response);

    // Queries
    TaskLog load(String taskId);
    List<TaskLog> list();
}
```

Each mutation method takes the parent's ID to place the new object in the hierarchy. For example, `runStarted` takes `stepId` so the store knows which step the run belongs to, and `turnStarted` takes `runId`.

The overload of `taskStarted(..., parentTaskId, parentStepId, iterationIndex)` is what loop/branch sub-tasks use so a `TaskLog` can point back at its parent step and iteration index.

### Reconstitution Constructors

`TaskLog`, `StepLog`, and `TurnLog` each expose constructors that accept full state — id, timestamps, child collections, etc. — so a persistent store can round-trip an existing row back into a log object without stamping fresh `Instant.now()` values. Use these when implementing `load(taskId)` against a database:

```java
new TaskLog(taskId, taskName, plan, params,
            parentTaskId, parentStepId, iterationIndex, createdAt);

new StepLog(id, stepName, createdAt);

new TurnLog(id, index, messageId, request,
            responseId, response, toolResults,
            startedAt, completedAt);
```

## TaskStateStoreMemory

The default in-memory implementation. Stores `TaskLog` objects in a `ConcurrentHashMap` keyed by task ID. Uses ID-based lookup to navigate the hierarchy:

```java
var store = new TaskStateStoreMemory();

// After a task runs:
var taskLog = store.load("a3cc1bf2");

// Navigate by name
var stepLog = taskLog.step("research");

// Or navigate by ID
var stepLog = taskLog.findStepById("7e4f0a21");
var runLog  = taskLog.findRunById("b8d31c09");
var turnLog = taskLog.findTurnById("f1a20e5b");
```

## Querying Execution State

### Token Usage

Token usage is captured per-turn in `LlmResponse` and aggregates up the hierarchy — each level sums its children. The underlying record:

```java
record TokenUsage(long input, long output, long cacheRead, long cacheWrite, long webSearches)
```

Query at any level:

```java
var taskLog = store.load(taskId);

// Task-level totals
taskLog.inputTokens();
taskLog.outputTokens();
taskLog.cacheReadTokens();
taskLog.cacheWriteTokens();
taskLog.webSearchRequests();

// Step-level
var step = taskLog.step("research");
step.inputTokens();

// Run-level
var run = step.lastRun();
run.inputTokens();

// Or get the full record
var usage = taskLog.tokenUsage();  // TokenUsage record
```

### Inspecting Turns

Each turn captures the full LLM exchange:

```java
for (var turn : run.turns()) {

    // The LLM request (system prompt, user message, tool definitions)
    var request = turn.request();

    // The LLM response (text, tool calls, token usage, stop reason)
    var response = turn.response();

    // Tool results from this turn
    for (var result : turn.toolResults()) {
        System.out.println(result.toolName() + ": " + result.output());
    }
}
```

### Timing

`TaskLog`, `StepLog`, and `TurnLog` capture start and completion timestamps:

```java
var duration = Duration.between(stepLog.createdAt(), stepLog.completedAt());
```

## Custom Implementations

For durable storage, implement the full `TaskStateStore` interface against your database. The mutation methods are called synchronously on the executing thread — keep them fast. Heavy work (indexing, replication) should be done asynchronously.

```java
Agentican.builder()
        .taskStateStore(myDatabaseStore)
        .build();
```

If no store is provided, Agentican creates a `TaskStateStoreMemory`.

## Event Emission

The store itself does not emit events. `TaskStateStoreNotifying` is a decorator that wraps any `TaskStateStore` and fires `TaskListener` events after each mutation. Agentican applies this decorator automatically — you don't need to wrap your store manually.

See [Observability](observability.md) for the event system.
